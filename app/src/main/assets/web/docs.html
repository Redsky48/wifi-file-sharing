WiFi Share — HTTP API

Programmatic access to your phone's shared folder. Same auth, same endpoints the web UI and Windows companion use.

Authentication

If the phone has a PIN set, every request needs HTTP Basic auth. The username is ignored — only the password matters.

Authorization: Basic <base64("user:<PIN>")>

If no PIN is set, requests work without the header. Public paths (/, /docs, /static/*, /favicon.ico, /download/*) never require auth.

Rate limit: 5 failed PINs from one IP within 60 seconds → 30-second lockout. Use the correct PIN.

Send a file to the phone

The simplest endpoint — stream the file body straight in. Filename + folder come from the path query parameter.

PUT /api/files?path=<destination>

curl

curl -u user:123456 \
     --upload-file ./report.pdf \
     "http://192.168.1.179:8080/api/files?path=Inbox/report.pdf"

PowerShell

$pin  = "123456"
$file = "C:\Users\me\report.pdf"
$dest = "Inbox/report.pdf"

$auth = [Convert]::ToBase64String([Text.Encoding]::UTF8.GetBytes("user:$pin"))
Invoke-RestMethod `
  -Uri "http://192.168.1.179:8080/api/files?path=$([Uri]::EscapeDataString($dest))" `
  -Method PUT `
  -Headers @{ Authorization = "Basic $auth" } `
  -InFile $file

Response (201 Created):

{
  "saved": "report.pdf",
  "path": "Inbox/report.pdf",
  "size": 248301
}

If a file with the same name exists, the server auto-renames: report.pdf → report (1).pdf. The actual saved name is in the response's saved field.

Files API

GET /api/files?path=<folder>

List a folder. Empty path = root. Response:

{
  "path": "Inbox",
  "entries": [
    {"name": "Pics",   "type": "folder", "modified": 1715200000000},
    {"name": "a.jpg",  "type": "file",   "size": 12345,
     "mime": "image/jpeg", "modified": 1715204000000}
  ],
  "allowUploads": true,
  "allowDelete":  false
}
PUT /api/files?path=<destination>

Upload a single file with the raw body. See above. Returns 201 Created.

POST /api/upload?path=<folder>

Multipart upload of one or more files. Use this from browser forms. Each form field with a filename gets saved. The path param is the destination folder (default: root).

curl -u user:PIN \
     -F "file=@report.pdf" \
     -F "file=@photo.jpg" \
     "http://phone:8080/api/upload?path=Inbox"
GET /api/download?path=<file>[&inline=1]

Download a file. inline=1 makes the browser display it instead of saving. Honours HTTP Range headers — good for video/audio streaming.

DELETE /api/delete?path=<file>

Delete a file or folder. Requires the "Allow delete" toggle in phone Settings.

Folder operations

POST /api/mkdir?path=<folder>

Create a folder (and any missing parents) under the shared root. Idempotent — succeeds if it already exists.

curl -X POST -u user:PIN "http://phone:8080/api/mkdir?path=Projects/2026/Q3"
POST /api/move?from=<src>&to=<dst>

Rename a file or folder. Source and destination must share the same parent folder (cross-folder moves use /api/copy + /api/delete).

curl -X POST -u user:PIN \
  "http://phone:8080/api/move?from=draft.txt&to=final.txt"
POST /api/copy?from=<src>&to=<dst>

Server-side duplicate of a file (no PC bandwidth). Auto-renames on name collision. Returns 201 with the actual saved path.

curl -X POST -u user:PIN \
  "http://phone:8080/api/copy?from=Inbox/a.jpg&to=Pics/a.jpg"

Metadata & search

GET /api/file?path=<path>

Stat a single file or folder (size, mime, modified). Cheaper than listing a parent folder.

GET /api/search?q=<query>[&path=<subroot>][&limit=200]

Recursive case-insensitive substring search. Returns up to limit matches (default 200, max 1000) plus a truncated boolean.

curl -u user:PIN "http://phone:8080/api/search?q=invoice&limit=20"
GET /api/storage

Disk usage on the volume that holds the shared folder.

{ "total": 256000000000, "used": 184000000000, "available": 72000000000 }

System

GET /api/health public — no PIN

Liveness probe. Returns {"status":"ok"} when the server is running. Use this in monitoring / load balancers.

GET /api/info public — no PIN

Server identity and capabilities — useful for discovery so the client can adapt before sending the PIN.

{
  "name": "WiFi Share",
  "version": "0.5.0",
  "device": "Pixel 9 Pro",
  "manufacturer": "Google",
  "sdk": 34,
  "authRequired": true,
  "allowUploads": true,
  "allowDelete": false
}
GET /api/clients

List recently active clients (visible in the phone's "Connected" card).

POST /api/notify

Push a custom system notification to the phone. Body is JSON. Great for CI hooks ("Build green"), home automations, monitoring alerts.

curl -X POST -u user:PIN \
     -H "Content-Type: application/json" \
     -d '{"title":"Build done","text":"All tests green ✓"}' \
     http://phone:8080/api/notify
GET /api/queue/{clientId}

For companion apps: poll for files the phone wants to push to this client. Returns the next item metadata or 204 No Content. Companions register first via POST /api/clients/register.

GET /api/queue/{clientId}/{itemId}

Download a queued item's body.

DELETE /api/queue/{clientId}/{itemId}

Acknowledge (ack) a queued item — removes it from the phone's queue. Call this after the body has been saved on the client.

Screen cast

Mirror the phone's screen to any browser on the LAN. Video only — no audio, by design. Start it from the phone app's Cast screen to browser button; the system permission dialog appears the first time.

GET /screen

Viewer HTML page. Open it in any LAN browser. Auto-detects whether the cast is on; full-screen toggle in the header.

GET /api/screen

Raw MJPEG stream (multipart/x-mixed-replace; boundary=wifishareframe). Drop it straight into an <img> tag:

<img src="http://phone:8080/api/screen">
GET /api/screen.jpg

One-shot latest JPEG frame — handy for thumbnails or shell scripts:

curl -u user:PIN http://phone:8080/api/screen.jpg -o latest.jpg
GET /api/screen/status

JSON: { running, frameCount, width, height }. Tail this to see if cast is live.

Remote control — touch / drag / keys

Once the phone user has enabled WiFi Share remote input under Settings → Accessibility, your scripts and AI agents can drive the phone via these endpoints. All coords are normalized 0..1 fractions — (0,0) top-left, (1,1) bottom-right of the actual phone display. The server multiplies by real screen pixels for you.

POST /api/input/tap?nx=0..1&ny=0..1

Single tap at the given fraction of the display.

curl -u user:PIN -X POST "http://phone:8080/api/input/tap?nx=0.5&ny=0.5"
POST /api/input/swipe?nx1=&ny1=&nx2=&ny2=&duration=

One-shot swipe A → B over duration ms (default 250). Good for fixed gestures (e.g. swipe-to-dismiss).

POST /api/input/drag?phase=start|move|end|cancel&nx=&ny=&duration=

Continuous drag — the proper way to do scroll, pan, and any long motion. Uses Android Accessibility's continueStroke so the finger never lifts between segments.

# Open the drag at the starting point
curl -u user:PIN -X POST "http://phone:8080/api/input/drag?phase=start&nx=0.5&ny=0.7"
# Stream move calls (throttle to ~30 ms; duration 35 keeps each segment snappy)
for i in 0.65 0.60 0.55 0.50 0.45 0.40; do
  curl -u user:PIN -X POST "http://phone:8080/api/input/drag?phase=move&nx=0.5&ny=$i&duration=35"
  sleep 0.03
done
# Lift the finger
curl -u user:PIN -X POST "http://phone:8080/api/input/drag?phase=end&nx=0.5&ny=0.4"
POST /api/input/scroll?nx=&ny=&ndx=&ndy=

Short fling centered on (nx, ny) by normalized delta (ndx, ndy). Convenience for wheel-style scrolling — use /api/input/drag for continuous motion.

POST /api/input/key?action=back|home|recents|notifications|quick_settings|lock|power

System global actions. Most useful: back, home, recents (app switcher).

GET /api/input/status

Returns {available: bool, hint: ...}. Hit this first — if available is false the user hasn't enabled accessibility yet and your inputs will return 503.

Agent control loop

Minimal Python agent that views + drives:

import requests, time
PHONE = "http://192.168.1.179:8080"
AUTH  = ("user", "123456")

# 1. Confirm the phone is casting + accepting input
s = requests.get(f"{PHONE}/api/screen/status", auth=AUTH).json()
assert s["running"], "ask user to start cast"
assert s["inputReady"], "ask user to enable Accessibility"

# 2. Pull a fresh frame to look at
img = requests.get(f"{PHONE}/api/screen.jpg", auth=AUTH).content
open("frame.jpg","wb").write(img)

# 3. Tap a button (model decides 0..1 coords from the image)
requests.post(f"{PHONE}/api/input/tap", params={"nx":0.5,"ny":0.92}, auth=AUTH)
time.sleep(0.5)

# 4. Smooth scroll down the page
requests.post(f"{PHONE}/api/input/drag", params={"phase":"start","nx":0.5,"ny":0.8}, auth=AUTH)
for y in [0.7, 0.6, 0.5, 0.4, 0.3]:
    requests.post(f"{PHONE}/api/input/drag",
        params={"phase":"move","nx":0.5,"ny":y,"duration":35}, auth=AUTH)
    time.sleep(0.03)
requests.post(f"{PHONE}/api/input/drag", params={"phase":"end","nx":0.5,"ny":0.3}, auth=AUTH)

Push events

Subscribe to a live stream of every interesting event the phone emits — file transfers, queue changes, client connect/disconnect, custom notifications. Two transports deliver the same envelopes; pick whichever your client supports.

WS /ws/events

WebSocket. Browsers and JS clients should connect with credentials in the URL (Basic auth on WS handshakes from fetch isn't supported). Server sends a hello frame on connect and a ping every 20s.

// Browser / Node
const ws = new WebSocket("ws://user:PIN@phone:8080/ws/events");
ws.onmessage = e => console.log(JSON.parse(e.data));
GET /api/events

Server-Sent Events — curl- and shell-friendly. Tail it live with:

curl -N -u user:PIN http://phone:8080/api/events

Sample frame:

data: {"type":"transfer.uploaded","ts":1762345678901,"seq":42,"data":{"name":"vacation.jpg"}}

Event types

transfer.uploaded    {name}          file uploaded to the phone
transfer.downloaded  {name}          file downloaded from the phone
transfer.deleted     {name}          file deleted via /api/delete
queue.changed        {count, items}  pending-upload queue changed
client.connected     {clientId,name,address}    companion registered
client.disconnected  {clientId,name,address}    inactivity-pruned (~5 min)
notification         {title, body}   /api/notify was called

Every envelope carries a monotonic seq field — agents that reconnect can spot dropped events by looking for gaps.

GET /api/manifest

Structured JSON describing every endpoint above plus the list of event types — fetch one URL and an AI agent can self-discover the API surface.

More examples

Python (requests)

import requests

phone = "http://192.168.1.179:8080"
auth  = ("user", "123456")  # PIN as password

# List files
r = requests.get(f"{phone}/api/files", auth=auth)
for entry in r.json()["entries"]:
    print(entry["type"], entry["name"])

# Send a file (streaming, no multipart)
with open("report.pdf", "rb") as f:
    r = requests.put(
        f"{phone}/api/files",
        params={"path": "Inbox/report.pdf"},
        data=f,
        auth=auth,
    )
print(r.json())

JavaScript (fetch)

const phone = "http://192.168.1.179:8080";
const headers = { Authorization: "Basic " + btoa("user:123456") };

// Send a Blob
const blob = new Blob(["hello, phone!"], { type: "text/plain" });
const url = `${phone}/api/files?path=` + encodeURIComponent("Inbox/note.txt");
const resp = await fetch(url, { method: "PUT", headers, body: blob });
console.log(await resp.json());

Plain HTTP (raw socket / Postman / etc.)

PUT /api/files?path=Inbox/note.txt HTTP/1.1
Host: 192.168.1.179:8080
Authorization: Basic dXNlcjoxMjM0NTY=
Content-Type: text/plain
Content-Length: 13

hello, phone!

Recipe: PowerShell folder watcher

Anything dropped in C:\PhoneInbox is auto-uploaded to the phone's Inbox folder.

$phone = "http://192.168.1.179:8080"
$pin   = "123456"
$watch = "C:\PhoneInbox"
$dest  = "Inbox"

$auth = "Basic " + [Convert]::ToBase64String([Text.Encoding]::UTF8.GetBytes("user:$pin"))
New-Item -ItemType Directory -Force -Path $watch | Out-Null

$fsw = New-Object IO.FileSystemWatcher $watch
$fsw.EnableRaisingEvents = $true

Register-ObjectEvent $fsw Created -Action {
    $path = $Event.SourceEventArgs.FullPath
    Start-Sleep -Milliseconds 200  # let the writer finish
    if (-not (Test-Path $path -PathType Leaf)) { return }
    $name = [IO.Path]::GetFileName($path)
    $url  = "$using:phone/api/files?path=" + [Uri]::EscapeDataString("$using:dest/$name")
    try {
        Invoke-RestMethod -Uri $url -Method PUT `
            -Headers @{ Authorization = $using:auth } `
            -InFile $path
        Write-Host "Sent $name"
    } catch {
        Write-Warning "Failed $name`: $($_.Exception.Message)"
    }
}

Write-Host "Watching $watch — drop files there to upload."
Write-Host "Press Ctrl+C to stop."
while ($true) { Start-Sleep -Seconds 1 }

Recipe: bash incoming watcher (Linux/macOS/WSL)

#!/usr/bin/env bash
PHONE="http://192.168.1.179:8080"
PIN="123456"
WATCH="$HOME/PhoneInbox"
DEST="Inbox"

mkdir -p "$WATCH"
inotifywait -m -e close_write --format '%f' "$WATCH" |
while read fname; do
    curl -sS -u "user:$PIN" \
         --upload-file "$WATCH/$fname" \
         "$PHONE/api/files?path=$DEST/$(printf %s "$fname" | jq -sRr @uri)" \
        && echo "Sent $fname"
done
Discovery tip: if you don't want to hard-code the phone's IP, advertise via mDNS. The phone announces _wifishare._tcp.local — Avahi / Bonjour / Windows DNS-SD can resolve it. Example: avahi-browse -tr _wifishare._tcp.

Versioning

This API is unversioned — endpoints have been stable since v0.5. The companion app, web UI, and these docs all hit the same surface.
